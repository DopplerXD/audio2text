# 本地音频转文字工具正式开发设计方案

版本：v1.1  
日期：2026-06-17

## 1. 项目概述

本项目用于在本地完成音频或视频文件的文字转写，并支持将识别结果导出为常用文本、字幕和结构化数据格式。项目面向个人本地使用场景，重点是安装简单、页面清晰、记录可追溯、文件路径可查看、数据不离开本机。

第一版采用轻量本地 Web 应用：Python 后端负责文件上传、音视频处理、模型识别、记录持久化和文件导出；前端使用原生 HTML、CSS、JavaScript 实现页面展示和交互。应用仍然只在本机运行，不建设复杂前后端工程，不引入用户系统、云端服务或任务队列。

## 2. 建设目标

### 2.1 核心目标

- 支持用户在本地页面上传单个音频或视频文件。
- 使用本地 MLX Whisper 模型完成语音转文字。
- 使用 HTML、CSS、JavaScript 实现一个简洁但更美观的本地页面。
- 展示完整文本、字幕分段、处理状态、耗时、生成文件路径。
- 保存每次处理记录，重启应用后仍可查看历史记录。
- 支持用户在页面中手动修改识别结果。
- 支持导出 TXT、Markdown、PDF、SRT、VTT、JSON。
- 支持一键打包导出全部结果为 ZIP。
- 全流程仅在本机处理，不上传音视频或识别文本。

### 2.2 非目标

第一版暂不实现以下能力：

- 实时麦克风转写。
- 多文件批量任务队列。
- 用户登录、权限和多用户隔离。
- 说话人识别和多人分离。
- LLM 总结、会议纪要、Agent 工作流。
- Vue、React 等前端框架。
- 桌面应用安装包。
- 云端模型或第三方转写 API。

## 3. 用户使用流程

```text
启动本地应用
    ↓
浏览器打开本地页面
    ↓
上传音频或视频文件
    ↓
选择语言、模型模式、时间戳选项
    ↓
点击“开始识别”
    ↓
页面展示处理状态、完整文本、字幕分段
    ↓
系统保存处理记录和文件路径
    ↓
用户按需编辑结果
    ↓
选择导出格式
    ↓
页面展示导出文件路径并提供下载
    ↓
后续可从历史记录重新打开结果
```

## 4. 总体架构

```text
浏览器页面
HTML + CSS + JavaScript
    ↓ fetch API
Python 本地后端
FastAPI / Uvicorn
    ↓
audio_utils.py 音视频预处理
    ↓
transcriber.py 调用 MLX Whisper
    ↓
models.py 统一结果结构
    ↓
storage.py SQLite 记录持久化
    ↓
exporters.py 导出 TXT / MD / PDF / SRT / VTT / JSON / ZIP
    ↓
data/records.db + outputs/ 本地文件
```

应用使用一个 Python 进程启动本地 HTTP 服务，同时托管静态前端页面和 API。前端不做独立构建，不使用 Node.js 工程；`index.html`、`styles.css`、`app.js` 直接由后端静态服务提供。

## 5. 技术选型

| 模块 | 技术 | 说明 |
| --- | --- | --- |
| 本地后端 | FastAPI + Uvicorn | 处理上传、识别、导出、记录查询，接口清晰 |
| 前端页面 | HTML + CSS + JavaScript | 无框架、无构建步骤，便于维护和美化 |
| 前端请求 | Fetch API | 调用本地 REST 接口 |
| 记录持久化 | SQLite | Python 标准库支持，适合保存本地历史记录和文件路径 |
| 语音识别 | mlx-whisper | 适配 Apple Silicon，本地推理路径简单 |
| 默认模型 | mlx-community/whisper-large-v3-turbo | 作为高质量默认模型，兼顾多语言识别效果 |
| 音视频处理 | FFmpeg | 统一处理 MP3、M4A、WAV、MP4、MOV 等输入格式 |
| PDF 导出 | ReportLab | 本地生成 PDF，支持注册中文字体 |
| 字幕导出 | 自定义 SRT/VTT 生成器 | 基于分段时间戳生成标准字幕文件 |
| 依赖管理 | venv 或 uv | 第一版保持简单，可优先使用 venv |

## 6. 页面设计

页面采用应用式布局，不做营销页。整体风格应简洁、安静、偏工具化，重点突出上传、识别状态、结果编辑、历史记录和文件路径。

### 6.1 布局结构

```text
顶部栏：项目名称、当前输出目录、运行状态
左侧栏：上传文件、识别配置、开始按钮、历史记录
主区域：当前任务状态、完整文本、字幕分段、导出结果
右侧信息栏：文件信息、生成路径、下载入口、错误信息
```

如果屏幕较窄，右侧信息栏可以折叠到主区域下方。第一版优先保证桌面端浏览器体验。

### 6.2 顶部栏

顶部栏展示：

- 应用名称：本地音频转文字。
- 当前输出目录：例如 `outputs/`。
- 后端状态：空闲、识别中、导出中、完成、失败。
- 简短版本号：例如 `v1.1`。

### 6.3 左侧栏：导入、配置与历史

左侧栏包含：

- 文件上传区域：支持拖拽和点击选择。
- 文件名、文件大小、文件类型展示。
- 语言选择：
  - 自动识别
  - 中文
  - 英文
  - 日语
  - 粤语
- 模型模式：
  - 快速
  - 标准
  - 高质量
- 词级时间戳开关：默认关闭。
- 开始识别按钮。
- 最近处理记录列表。

历史记录列表展示：

- 原始文件名。
- 处理状态。
- 创建时间。
- 识别耗时。
- 快捷操作：打开、导出、删除记录。

### 6.4 主区域：识别结果

主区域使用 Tab 或分段按钮组织结果：

#### 完整文本

- 多行文本编辑器展示完整转写文本。
- 用户可直接修改文本。
- 显示字数、分段数、识别语言。
- 提供保存修改按钮。

#### 字幕分段

- 表格展示每个分段的开始时间、结束时间、内容。
- 用户可修改分段文本。
- 第一版不要求在页面内精细调整时间戳，但时间字段应展示清楚。
- 支持由分段重新拼接完整文本。

#### 导出结果

- 展示已生成文件列表。
- 每个文件展示格式、文件名、绝对路径、文件大小、生成时间。
- 提供下载按钮。
- 提供复制路径按钮。
- 提供重新生成按钮。

### 6.5 右侧信息栏：文件与路径

右侧信息栏用于解决“文件生成在哪里”的问题，展示内容包括：

- 当前记录 ID。
- 原始文件路径。
- 临时音频路径。
- 输出目录路径。
- 已生成文件路径。
- 识别模型。
- 识别语言。
- 音频时长。
- 识别耗时。
- 最后更新时间。

路径较长时使用单行省略，悬停或点击后展示完整路径。第一版至少需要提供“复制路径”按钮，不强制实现“在 Finder 中打开”。

## 8. 数据模型

内部以识别记录为顶层对象，以字幕分段作为核心结果数据源。完整文本可以由分段内容拼接得到，也可以保留 Whisper 返回的完整文本作为初始值。用户编辑后，导出模块应优先使用页面当前状态，而不是重新读取原始识别结果。

```python
from dataclasses import dataclass, field


@dataclass
class Segment:
    id: int
    start: float
    end: float
    text: str
    words: list[dict] = field(default_factory=list)


@dataclass
class TranscriptionResult:
    filename: str
    language: str | None
    duration: float | None
    text: str
    segments: list[Segment]


@dataclass
class TranscriptionRecord:
    id: str
    original_filename: str
    original_path: str
    temp_audio_path: str | None
    output_dir: str
    status: str
    model_mode: str
    model_name: str
    language: str | None
    duration: float | None
    elapsed_seconds: float | None
    text: str
    segments: list[Segment]
    export_files: list[dict]
    error_message: str | None
    created_at: str
    updated_at: str
```

JSON 导出格式：

```json
{
  "id": "20260617-171900-meeting",
  "original_filename": "meeting.m4a",
  "language": "zh",
  "duration": 1832.5,
  "text": "完整识别文本",
  "segments": [
    {
      "id": 0,
      "start": 0.0,
      "end": 4.32,
      "text": "大家好，今天我们讨论一下本地语音识别工具。",
      "words": []
    }
  ],
  "export_files": [
    {
      "format": "txt",
      "filename": "meeting.txt",
      "path": "outputs/20260617-171900-meeting/meeting.txt",
      "size": 2048,
      "created_at": "2026-06-17T17:19:00+08:00"
    }
  ]
}
```

## 9. SQLite 持久化设计

第一版使用 SQLite 保存处理记录和导出文件索引，识别分段可以以 JSON 字符串保存，避免过早设计复杂表结构。

### 9.1 records 表

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | TEXT PRIMARY KEY | 记录 ID |
| original_filename | TEXT | 原始文件名 |
| original_path | TEXT | 原始上传文件路径 |
| temp_audio_path | TEXT | 转码后的临时音频路径 |
| output_dir | TEXT | 当前记录的输出目录 |
| status | TEXT | pending、running、completed、failed |
| model_mode | TEXT | 快速、标准、高质量 |
| model_name | TEXT | 实际模型名 |
| language | TEXT | 识别语言 |
| duration | REAL | 音频时长 |
| elapsed_seconds | REAL | 识别耗时 |
| text | TEXT | 完整文本 |
| segments_json | TEXT | 分段 JSON |
| error_message | TEXT | 错误信息 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 9.2 export_files 表

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PRIMARY KEY AUTOINCREMENT | 文件索引 ID |
| record_id | TEXT | 所属记录 ID |
| format | TEXT | txt、md、pdf、srt、vtt、json、zip |
| filename | TEXT | 文件名 |
| path | TEXT | 文件路径 |
| size | INTEGER | 文件大小 |
| created_at | TEXT | 生成时间 |

## 10. API 设计

前端通过 Fetch API 调用本地接口。所有接口只监听本机地址，例如 `127.0.0.1:7860`。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/health` | 获取后端状态 |
| `POST` | `/api/transcriptions` | 上传文件并开始识别 |
| `GET` | `/api/transcriptions` | 获取历史记录列表 |
| `GET` | `/api/transcriptions/{id}` | 获取单条记录详情 |
| `PATCH` | `/api/transcriptions/{id}` | 保存编辑后的文本和分段 |
| `DELETE` | `/api/transcriptions/{id}` | 删除记录，可选删除关联文件 |
| `POST` | `/api/transcriptions/{id}/exports` | 生成指定格式导出文件 |
| `POST` | `/api/transcriptions/{id}/exports/all` | 生成全部格式并打包 ZIP |
| `GET` | `/api/files/{file_id}` | 下载导出文件 |

第一版可以同步执行识别，请求完成后直接返回记录详情。长音频识别时页面显示加载状态。后续如果需要更好的进度显示，再扩展为后台任务和轮询。

## 11. 核心模块设计

### 11.1 app.py

职责：

- 创建 FastAPI 应用。
- 挂载 `static/` 静态目录。
- 初始化 `uploads/`、`temp/`、`outputs/`、`data/` 目录。
- 初始化 SQLite 数据库。
- 启动本地服务。

### 11.2 api.py

职责：

- 接收上传文件和用户参数。
- 调用音视频预处理、识别、存储和导出函数。
- 返回前端需要的 JSON 数据。
- 将内部异常转成清晰的接口错误信息。

### 11.3 static/index.html

职责：

- 定义页面结构。
- 包含顶部栏、左侧栏、主内容区、右侧信息栏。
- 不写复杂业务逻辑。

### 11.4 static/styles.css

职责：

- 控制整体布局、颜色、间距、按钮、表格、状态标签。
- 保证常用桌面浏览器下界面清晰。
- 路径、文件名、错误信息等长文本不撑破布局。

建议视觉规则：

- 背景使用浅灰色，主内容使用白色面板。
- 主按钮使用稳定的蓝色或绿色。
- 状态标签使用不同颜色区分运行中、完成、失败。
- 历史记录列表保持紧凑，方便快速扫描。
- 文件路径使用等宽字体展示。

### 11.5 static/app.js

职责：

- 处理文件选择、表单参数、按钮点击。
- 调用 `/api/*` 接口。
- 渲染历史记录、当前记录详情、分段表格和导出文件列表。
- 管理页面 loading、错误提示和状态标签。
- 实现复制路径、保存编辑、触发导出、下载文件等交互。

前端状态可以保持为一个简单对象：

```javascript
const state = {
  records: [],
  currentRecord: null,
  isTranscribing: false,
  isExporting: false
};
```

### 11.6 transcriber.py

职责：

- 根据页面模型模式映射实际 MLX Whisper 模型。
- 调用 `mlx_whisper.transcribe()`。
- 处理语言参数：自动识别时不传语言，固定语言时传对应代码。
- 将 Whisper 原始结果转换为统一的 `TranscriptionResult`。
- 捕获模型加载、文件读取、识别失败等异常并返回清晰错误信息。

模型模式映射：

| 页面模式 | 实际模型 |
| --- | --- |
| 快速 | mlx-community/whisper-small |
| 标准 | mlx-community/whisper-medium |
| 高质量 | mlx-community/whisper-large-v3-turbo |

如快速或标准模型在本地不可用，可先只实现高质量模式，并在页面上保留模型选择接口。

### 11.7 audio_utils.py

职责：

- 检查 FFmpeg 是否可用。
- 校验上传文件扩展名。
- 保存原始上传文件到 `uploads/{record_id}/`。
- 必要时将输入转为 16kHz 单声道 WAV。
- 获取音频时长。
- 清理过期临时文件。

支持的输入格式：

- 音频：`.mp3`、`.m4a`、`.wav`、`.aac`、`.flac`
- 视频：`.mp4`、`.mov`、`.mkv`

### 11.8 exporters.py

职责：

- 将当前页面结果导出为不同格式。
- 每条记录使用独立输出目录：`outputs/{record_id}/`。
- 每次导出生成文件记录，避免用户找不到文件。
- 保证导出内容与用户编辑后的页面内容一致。

导出规则：

- TXT：只输出完整文本。
- Markdown：输出文件信息、完整文本、可选分段表。
- PDF：输出文件名、识别语言、音频时长、生成时间、完整文本。
- SRT：按字幕编号和 `HH:MM:SS,mmm` 时间格式输出。
- VTT：以 `WEBVTT` 开头，使用 `HH:MM:SS.mmm` 时间格式。
- JSON：输出完整结构化数据。
- ZIP：包含全部导出格式。

### 11.9 storage.py

职责：

- 初始化 SQLite 表结构。
- 新建、更新、查询、删除识别记录。
- 保存导出文件路径、大小和生成时间。
- 将分段列表序列化为 JSON 存入数据库。
- 从数据库读取记录时还原为前端需要的结构。

## 12. 关键处理规则

### 12.1 记录 ID 与目录

每次上传生成一个唯一记录 ID，建议格式：

```text
YYYYMMDD-HHMMSS-安全文件名
```

示例：

```text
20260617-171900-meeting
```

对应目录：

```text
uploads/20260617-171900-meeting/
temp/20260617-171900-meeting/
outputs/20260617-171900-meeting/
```

### 12.2 文件路径展示

后端保存相对路径和绝对路径均可，但接口返回时建议同时包含：

- `path`：相对项目根目录路径，便于页面展示。
- `absolute_path`：绝对路径，便于复制后在系统中定位。

页面默认展示相对路径，复制按钮复制绝对路径。

### 12.3 时间格式

内部统一使用秒数浮点值。导出时再转换为目标格式：

- SRT：`00:00:04,320`
- VTT：`00:00:04.320`
- 页面展示：`00:00:04.320`

### 12.4 文本一致性

用户编辑完整文本后：

- TXT、Markdown、PDF 使用编辑后的完整文本。

用户编辑字幕分段后：

- SRT、VTT、JSON 使用编辑后的分段内容。
- 如果完整文本为空，可以由分段文本重新拼接。

用户点击“保存修改”后：

- 后端更新 `records.text`。
- 后端更新 `records.segments_json`。
- 后端更新 `updated_at`。

### 12.5 错误处理

需要给用户明确提示的错误：

- 未安装 FFmpeg。
- 上传文件为空或格式不支持。
- 文件无法被 FFmpeg 读取。
- 模型下载或加载失败。
- 识别过程中断。
- 导出文件写入失败。
- 数据库写入失败。
- 下载文件不存在。

错误提示应显示在页面状态区，不输出大段 Python 堆栈。

### 12.6 本地隐私

- 不调用云端识别 API。
- 不上传文件到外部服务。
- 原始文件默认保存在项目 `uploads/` 目录。
- 临时文件默认保存在项目 `temp/` 目录。
- 导出文件默认保存在项目 `outputs/` 目录。
- 历史记录默认保存在项目 `data/records.db`。
- 后续可以增加“一键清理临时文件”和“打开输出目录”按钮。

## 13. 安装与运行

基础环境：

## 15. 验收标准

第一版完成后，应满足以下标准：

- 能在本地通过 `python app.py` 启动页面。
- 页面由 HTML、CSS、JavaScript 实现，不依赖 Gradio。
- 能上传一个常见音频文件并完成识别。
- 能上传一个常见视频文件并提取音频完成识别。
- 能展示完整文本。
- 能展示至少包含开始时间、结束时间、文本三列的字幕分段。
- 能保存识别记录，重启应用后仍可查看历史记录。
- 能展示原始文件路径、输出目录和已生成文件路径。
- 修改页面文本后，保存记录并在导出文件中体现修改结果。
- TXT、Markdown、PDF、SRT、VTT、JSON 均能成功生成。
- ZIP 导出包含全部格式文件。
- 导出文件列表中可以看到格式、文件名、路径、大小和生成时间。
- 未安装 FFmpeg、格式不支持、识别失败时有明确页面提示。
- 处理过程不依赖外部业务服务，不需要云端 API。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 首次加载模型较慢 | 用户以为程序卡住 | 页面显示“模型加载中”和耗时 |
| 模型文件较大 | 初次使用等待时间长 | 文档中说明首次运行会下载模型 |
| FFmpeg 未安装 | 无法处理部分格式 | 启动或识别前检查并提示安装命令 |
| PDF 中文乱码 | PDF 不可读 | 注册 macOS 中文字体，如苹方或华文字体 |
| 长音频识别耗时较长 | 页面等待时间长 | 第一版先提示耗时，后续再做切分和进度优化 |
| 原生 JS 状态管理混乱 | 页面交互变难维护 | 使用单一 `state` 对象和少量渲染函数 |
| 文件路径过长 | 页面布局被撑开 | 路径区域使用等宽字体、省略显示和复制按钮 |
| SQLite 记录与文件不一致 | 历史记录失效 | 页面标记文件缺失，并提供删除无效记录 |
